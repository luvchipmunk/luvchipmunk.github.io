## 추가 AWS 컴퓨팅 서비스

일반적인 EC2 과정은.

인스턴스(가상 서버)를 프로비저닝 -> 사용자 코드 업로드 -> 실행되는 App 인스턴스를 관리

이지만,

- 서버리스 컴퓨팅

서버리스(코드가 서버에서 실행되는 서버를 프로비저닝하거나 관리할 필요x)컴퓨팅의 경우는 사용자가 App에 집중하게한다. 자동으로 확장가능한 유연성 역시 가진다.

(처리량 및 메모리양을 조절하여 App 용량 조절 가능)

대표적으로 AWS Lambda 서비스가 있다.

![이벤트기반 프로세스](/images/2026-01-04-AWS4/이벤트기반 프로세스.png)

AWS Lambda는 서버리스 컴퓨팅 이므로, 서버를 프로비저닝 하거나 관리할 필요 없이 코드를 실행할 수 있는 서비스이다.

1. 코드를 Lambda에 업로드

2. AWS 서비스, App , 또는 HTTP 엔드포인트와 같은 이벤트 소스에서 트리거(Trigger)되도록 코드를 설정.

3. 트리거 된 경우에만 Lambda는 코드를 실행한다.

4. 사용한 컴퓨팅 시간에 대해서만 요금을 지불.

   



- 컨테이너식 App을 빌드하고 실행할 수 있다.(컨테이너 오케스트레이션 도구)

컨테이너(개발자의 입장에서 App 환경이 배포와 상관없이 일관되게 유지되기를 원하는데, 이때 컨테이너식 접근방식을 사용.)는 App의 코드와 종속성을 하나의 객체를 패키징하는 표준 방식을 제공.

보안성, 안정성, 확장성 요구 사항이 매우 중요한 프로세스 및 워크플로에도 컨테이너를 사용.

1. Amazon ECS(Amazon Elastic Cotainer Service)

AWS에서 컨테이너식 App을 실행하고 확장할 수 있는 확장성이 뛰어난 고성능 컨테이너 관리 시스템이다.

ECS는 Docker 컨테이너를 지원.

2. Amazon EKS(Amazon Elastic Kubernetes Service)

AWS에서 Kubernetes를 실행하는 데 사용할 수 있는 완전 관리형 서비스.

cf) AWS Fargate

Fargate는 컨테이너용 서버리스 컴퓨팅 엔진으로, ECS와 EKS에서 작동함.

이 경우 서버를 프로비저닝하거나 관리할 필요가 없고, Fargate가 자동으로 서버 인프라를 관리한다. 컨테이너를 실행하는데 필요한 리소스에 대해서만 비용을 지불하면 된다.

++ 컴퓨팅 서비스에 대한 결론.

1. 기존 App 호스팅 + OS에 대한 전체 액세스를 원한다 -> EC2 사용.
2. 단기 실행 함수 호스팅 또는 서비스 중심 / 이벤트 기반 App을 호스팅하며

서버를 프로비저닝 또는 서버를 관리하지 않고 싶다. -> AWS Lambda

3. AWS에서 Docker 컨테이너 기반 워크로드를 실행하고 싶다.

-> AWS ECS / AWS EKS 중에서 선택.

-> 이후 이 컨테이너를 EC2 인스턴스에서 사용자가 관리 하고싶다면 EC2에서 실행

or EC2 인스턴스를 관리할 필요가 없거나 기본 OS에 액세스할 필요가 없으면 AWS Fargate라는 컴퓨팅 플랫폼을 사용하기.

정리)

클라우드 컴퓨팅 -> 종량 과금제 요금으로 필요에따라 IT리소스 제공(온디맨드 제공)

————————————————————

EC2 - Ec2 인스턴스(가상 서버)를 동적으로 가동 및 중단 가능.

=> 시작할 때, 용도에 맞춰 인스턴스 패밀리 선택 (하드웨어 결정)

=> 인스턴스 규모를 조정하여 수직 / 수평으로 확장가능하다.

여기서 수평 확장을 하는데 자동 수평 확장 기능인 EC2 Auto Scaling 도구를 사용.

이제 수평으로 확장한 후에는 수신하는 트래픽을 인스턴스에 분산하는데, ELB 사용.

cf ) EC2 인스턴스는 다양한 요금제를 제공한다.

(온디맨드 / Savings Plan / 예약 인스턴스 등등)

- 메시징 서비스와 컴퓨팅 서비스 역시 다양하게 제공한다.
