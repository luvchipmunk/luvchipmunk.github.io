## EC2

Amazon Elastic Compute Cloud(Amazon EC2)는 안전하고 크기 조정이 가능한 컴퓨팅 용량을 EC2 인스턴스로 클라우드에서 제공.

온 프레미스 리소스를 사용할 경우,

\> 하드웨어 구매에 비용이 많이듬. 서버 배달 및 설치에 기간이 많이 듬.

최종적으로 사용 여부와 상관없이 무조건 보유해야한다.

이에 비해 EC2 인스턴스를 사용할 경우, AWS 클라우드에서 가상 서버를 사용하여 App을 실행할 수 있다.

\> 몇 분이면 EC2 인스턴스를 프로비저닝하고 시작할 수 있다.

워크로드 실행완료시 인스턴스 사용중지.

사용한 서버 용량에 대해서만 비용 지불.

AWS가 데이터 센터를 구축 중이므로, 보호중이다.

EC2는 가상화 기술을 사용하여 AWS에서 관리하는 물리 호스트 시스템에서 실행됨.

(EC2의 경우는 인스턴스 가동시, 전체 호스트를 소요하지 않아도 되나 다른 여러 인스턴스와 공유하여 호스트를 가질 수 있다. - “가상머신”)

-> 호스트 머신에서 실행하는 하이퍼 바이저가 가상머신 간의 기본 물리적 리소스 공유를 책임진다.

‘멀티 테넌시 (-> 가상 컴퓨터간 하드웨어 공유) 조정을 책임진다’는 의미.

하드웨어 리소스를 공유하는 가상 머신을 서로 분리하는 일을 책임진다.

다른 인스턴스까지 인식하지 못할 정도로 안전하게 분리되어 있다.

EC2를 프로비저닝 하는데에 있어, OS를 윈도우나 리눅스를 선택할 수 있다.

EC2 인스턴스 수천개를 온디맨드로 프로비저닝 할 수 있다.(다양한 App 가동할 수 있다.)

OS가 아닌 인스턴스에서 실행할 S/W로 구성 가능.(ex-내부 비즈니스/Web App, DB 등등)

EC2 인스턴스 크기조정도 가능(더 많은 리소스 필요시 제공 가능. 이를 ‘인스턴스 수직확장’)

심지어 서버에 전달할 요청 유형과 공개/비공개 액세스 여부 결정 가능.

EC2 네트워킹 측면도 제어가능.

∴ 인스턴스를 시작하고 , 구성 인스턴스가 포함된 템플릿을 선택하여 시작.

인스턴스 시작 준비할때, 인스턴스로 들어오는 네트워크 트래픽 제어할 보안 설정 지정.



EC2 인스턴스 유형 종류 - [참조](https://aws.amazon.com/ko/ec2/instance-types/)

AWS 환경에서 배포할 수 있는 다양한 유형의 인스턴스가 있다.

EC2 인스턴스 유형은 인스턴스 패밀리로 분류/그룹화 된다.

특정 유형의 작업에 최적화 하기 위함임.

인스턴스 유형에 따라 리소스 제공.

App을 위한 적절한 리소스 조합 선택가능.

-> 워크로드 / App의 구체적 요구사항 고려.

종류는 범용 / 컴퓨팅 최적화 / 메모리 최적화 / 액셀러레이티드 컴퓨팅 / 스토리지 최적화

EC2 요금제

1. 온디맨드 -> 쓴 기간만큼 비용
2. Saving plans(약정인듯) -> 일정한 사용량 약정.(1-3년 동안 시간당 비용청구)
3. 예약인스턴스 -> 꾸준한 상태의 워코르드나 사용량 예측이 가능한 워크로드에 적합.
4. 스팟 인스턴스 -> 여분의 EC2 컴퓨팅 용량을 온디맨드 대비 최대 90% 할인

=> AWS가 언제든지 인스턴스를 회수한 다음, 작업 마무리 하라고 경고하곤함.

(워크로드에서 중단을 허용할 수 있는 것인지 확인해야함.) ex - 배치 워크로드.

EC2 규모 조정 - EC2 Auto Scaling

‘장애는 언제든지 발생하니, 장애에 대비한 설계로 무장애를 실현하기’

온프레미스 데이터 센터 구축에 있어서는 사용량에 맞춘 하드웨어 구매 추정이 굉장히 어렵다. 피크치에 맞춰 너무 넉넉히 구매하면 유휴 리소스가 발생할 것이고, 평균에 맞춰 구매한다면 피크 상황 발생시 하드웨어 부족으로 서비스 제공이 어렵다.

이에 맞춰서 매시간 수요에 맞는 워크로드를 프로비저닝을 하게끔 하는 것이

EC2 Auto Scaling 기능이다.

-> 변화하는 App 수요에 따라 EC2 인스턴스를 자동으로 추가하거나 제거.

동적 조정 / 예측 조정 두 가지 접근 방식으로 사용할 수 있다.

1. 동적 조정 : 수요변화에 대응한다.
2. 예측 조정 : 예측된 수요에 따라 Ec2 인스턴스를 자동으로 예약

- 증가하는 수요 처리를 수직 확장 / 수평 확장을 통해 처리 한다.
- EC2 인스턴스 생성에 있어서 희망용량(최소/최대 용량) 설정을 한다.



Elastic Load Balancing(ELB)

Ec2 인스턴스 전체에 워크로드를 균일하게 분산하려면 어떻게 하는가?

요청처리를 위해 요청을 인스턴스로 라우팅할 방법이 필요하다. -> ‘로드 밸런싱’

로드 밸런서는 요청을 받은 이후, 처리할 인스턴스로 라우팅하는 App이다.

= 트래픽 분산

고성능 , 비용 효율성, 고가용성, 자동 확장 기능.

ELB(Elastic Load Balancing)는 주요 관리형 서비스 중 하나

개별 EC2 인스턴스 형태가 아닌, 리전 수준 구조를 가진다. (자동으로 고가용 서비스로 작동되도록함.)

\> 트래픽 증가시, 시간당 비용 변경없이 추가 처리량을 처리하도록 설계됐다.

=> 들어오는 aPP 트래픽을 EC2 인스턴스와 같은 여러 리소스에 자동으로 분산하는 서비스.

\> Auto Scaling 그룹으로 들어오는 트래픽의 단일 접점 역할을 한다.

들어오는 트래픽 양에 맞춰 EC2 인스턴스를 추가하거나 제거하므로, 이런 요청이 로드밸런서로 먼저 라우팅 된다.

그 다음 처리할 여러 리소스로 분산 된다.

때문에 하나의 인스턴스가 대량으로 워크로드를 처리할 필요가 없도록만든다.

ELB와 Auto Scaling은 별도 서비스지만 서로 연동하여 성능과 가용성을 둘다 챙긴다.



ELB는 외부 트래픽 뿐만아니라, 효율적인 백엔드 트래픽을 관리하는데 있어도 끊임없이 수요에 맞춰서 조절한다. 외부에서는 백엔드 인스턴스 수를 신경 쓸 필요 조차 없다.
